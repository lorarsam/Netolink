1. Contexto

La fintech quedó contenta con tu prototipo de banca digital... hasta que un usuario transfirió −$50.000 y su saldo subió. El código "funcionaba" cuando lo probaste a mano, pero nadie lo estaba probando de verdad. Ahora el equipo exige lo que exige cualquier equipo serio: una suite de tests unitarios que garantice que la lógica crítica del banco no se rompe con cada cambio.

En esta evaluación vas a tomar tu propio proyecto XBank de la Evaluación 1 y le vas a agregar tests unitarios con Vitest y React Testing Library. Nada de proyectos nuevos: se testea lo que ya construiste. Y sí, esto probablemente te obligue a refactorizar: el código difícil de testear casi siempre es código mal estructurado. Eso también es parte del aprendizaje.

Importante: los tests unitarios no tocan Firebase real. Nada de conectarse a Firestore ni crear usuarios de verdad en los tests. Todo lo que sea Firebase se reemplaza con mocks (vi.mock, vi.fn). Un test que depende de internet o de un servicio externo no es un test unitario.

2. Objetivos de aprendizaje

  • Escribir tests unitarios con Vitest siguiendo el patrón AAA (Arrange, Act, Assert).
  • Testear componentes React con React Testing Library desde la perspectiva del usuario: qué ve y qué hace, no cómo está implementado por dentro.
  • Simular eventos de usuario (escribir, hacer clic, enviar formularios) con userEvent.
  • Aislar la lógica de Firebase mediante mocks (vi.mock, vi.fn) para testear sin depender de servicios externos.
  • Refactorizar código para hacerlo testeable: extraer lógica pura, separar responsabilidades.
  • Interpretar un reporte de cobertura y entender qué mide (y qué no mide).

3. Descripción del trabajo

Debes agregar una suite de tests a tu XBank cubriendo tres niveles: lógica pura, componentes y servicios mockeados.

3.1 Requisitos (obligatorios)

RT1 · Setup del entorno de testing

Configurar Vitest + React Testing Library + jsdom en el proyecto existente. La suite completa debe correr con npm test y el reporte de cobertura con npm run coverage. Ambos scripts deben quedar en el package.json.

RT2 · Tests de lógica pura: validaciones de transferencia

Extrae las validaciones de transferencia a funciones puras (ej. src/utils/validaciones.js) y testéalas de forma exhaustiva. Casos mínimos:

  • Monto negativo, cero, no numérico y con decimales inválidos → rechazado.
  • Monto mayor al saldo disponible → rechazado.
  • Transferencia a uno mismo → rechazada.
  • Destinatario vacío o con formato de email inválido → rechazado.
  • Caso feliz: monto válido con saldo suficiente → aceptado.

Si tus validaciones hoy viven mezcladas dentro del componente, este es el momento de refactorizar. El componente llama a la función; la función se testea sola.

RT3 · Tests de componentes: formulario de transferencia

Con React Testing Library y userEvent, testea el comportamiento del formulario de transferencia:

  • Renderiza los campos y el botón de enviar.
  • Al ingresar un monto inválido y enviar, se muestra el mensaje de error correspondiente y no se llama al servicio de transferencia.
  • Al enviar datos válidos, se llama al servicio (mockeado) exactamente una vez y con los argumentos correctos.
  • Mientras la transferencia está "en curso", el botón queda deshabilitado (no hay doble submit).

RT4 · Tests de componentes: login e historial

  • Login: con campos vacíos no se llama al servicio de autenticación; si el servicio mockeado rechaza (credenciales inválidas), se muestra un mensaje de error al usuario.
  • Historial: dado un arreglo de movimientos (datos de prueba, sin Firestore), el componente los renderiza ordenados del más reciente al más antiguo, distingue envíos de recepciones, y muestra un estado vacío cuando no hay movimientos.

RT5 · Mock de la capa de servicios

Los módulos de src/services/ (o donde tengas tu lógica de Firebase) se mockean con vi.mock. Debes demostrar al menos: un mock que resuelve con éxito, un mock que rechaza con error, y verificación de llamadas con toHaveBeenCalledWith. Ningún archivo de test debe importar la configuración real de Firebase.

RT6 · Cobertura mínima

La suite debe alcanzar al menos 70% de cobertura de líneas en src/utils/ y en los componentes testeados (formulario de transferencia, login, historial). No se exige cubrir el 100% del proyecto: se exige cubrir bien lo crítico. En el README debes incluir una captura o resumen del reporte de cobertura.

3.2 Opcionales (bonificación, máx. +0,5 en la nota)

Bonus

  • Test del hook de suscripción: verificar que al desmontar el componente se llama la función unsubscribe (mockeada). Este es el test más interesante de toda la evaluación.
  • GitHub Actions: workflow que corre npm test en cada push. El badge verde en el README vale oro.
  • Tests parametrizados con it.each para la batería de validaciones.

4. Requisitos técnicos: aquí se juega la nota

4.1 Buenas prácticas de testing

  • Estructura AAA visible en cada test: preparar, ejecutar, verificar. Un test que hace las tres cosas revueltas es ilegible.
  • Nombres descriptivos que digan comportamiento: it('rechaza transferencias con monto negativo'), no it('test 3').
  • Un comportamiento por test. Si un test tiene 15 asserts sobre cosas distintas, son varios tests disfrazados de uno.
  • Tests independientes: cada test debe poder correr solo y en cualquier orden. Usa beforeEach para resetear mocks (vi.clearAllMocks()), no dependas del estado que dejó el test anterior.
  • Queries de RTL por accesibilidad primero: getByRole, getByLabelText, getByText. El getByTestId es el último recurso, no el primero.
  • Para código asíncrono usa findBy* / waitFor. Nada de setTimeout a mano dentro de un test.

4.2 Lo que NO se acepta

  • Tests que se conectan a Firebase real (aunque sea un proyecto de prueba).
  • Tests triviales para inflar cobertura: expect(true).toBe(true), testear que un título renderiza un título, o snapshots gigantes sin criterio.
  • Tests que dependen de detalles internos de implementación (leer el estado interno del componente en vez de su comportamiento visible).
  • Tests comentados o con it.skip para "que pase la suite". Si un test falla, se arregla el test o se arregla el código.

4.3 Stack obligatorio

Vitest como runner (nativo de Vite, no Jest) · @testing-library/react + @testing-library/user-event · jsdom como entorno · @vitest/coverage-v8 para el reporte. Los mocks se hacen con las herramientas de Vitest (vi.mock, vi.fn, vi.spyOn); no se permiten emuladores de Firebase para esta evaluación.

5. Rúbrica de evaluación

CriterioDescripciónPuntaje
Tests de lógica pura RT2: validaciones extraídas y testeadas de forma exhaustiva, incluyendo casos borde y caso feliz. 20%
Tests de componentes RT3 y RT4: interacción simulada con userEvent, verificación de mensajes de error, doble submit y renderizado condicional. 25%
Mocking RT5: capa de servicios correctamente aislada con vi.mock, casos de éxito y de error, verificación de llamadas. Cero conexiones a Firebase real. 20%
Calidad de los tests AAA, nombres descriptivos, independencia entre tests, queries accesibles, manejo correcto de asincronía. Cobertura mínima cumplida sin tests basura. 20%
Defensa Explicas tus tests y decisiones. Preguntas tipo: ¿por qué mockeas el servicio en vez de usar Firebase real? ¿qué diferencia hay entre getBy y findBy? ¿qué garantiza (y qué no garantiza) tu 70% de cobertura? ¿qué tuviste que refactorizar para poder testear? 15%

Descuentos: la suite no corre con npm test siguiendo el README (−2,0 puntos en la nota final) · tests que se conectan a Firebase real (−1,0) · tests triviales o de relleno para inflar cobertura (−0,5) · tests dependientes entre sí o con it.skip para ocultar fallas (−0,5).

6. Uso de IA

Mismas reglas de siempre, con una advertencia extra: la IA es especialmente buena generando tests que parecen buenos pero testean puras trivialidades. Tu criterio es el filtro.

  • Todo test que entregas lo debes poder explicar: qué comportamiento verifica y por qué importa.
  • Si la IA te genera 40 tests, tu pega es decidir cuáles aportan y cuáles son ruido. Cantidad no es calidad.
  • En el README, en la sección de uso de IA, agrega qué le pediste para los tests y al menos un ejemplo de un test generado que tuviste que corregir o descartar (y por qué).

7. Formato de entrega

  • Mismo repositorio de la Evaluación 1, con commits nuevos que muestren el trabajo de esta etapa (incluidos los refactors). Si tuviste observaciones en la Eval. 1, este es el momento de corregirlas de pasada.
  • README.md actualizado con: instrucciones para correr los tests (npm test, npm run coverage), resumen o captura del reporte de cobertura, qué refactorizaste para hacer el código testeable, y la sección de uso de IA.
  • La suite completa debe pasar en verde con npm install + npm test siguiendo el README. No se exige deploy.
  • Entrega en la plataforma del curso: link al repositorio, antes de la fecha límite indicada en clases.

Consejo final: parte por RT2 (las validaciones puras). Es lo más fácil de testear y te obliga al refactor que después te salva en RT3. Si el día 3 ya tienes las validaciones extraídas y verdes, los tests de componentes salen solos. Si el día 6 recién estás peleando con la configuración de jsdom, se te vino la noche. Planifica.
