1. Contexto

Una fintech te contrató para construir el prototipo de su banca digital. No necesitan algo gigante: necesitan una interfaz simple, correcta y reactiva, donde los datos se actualicen en tiempo real sin que el usuario tenga que refrescar la página. Tu trabajo no es solo que "funcione": es demostrar que sabes estructurar una aplicación React con buenas prácticas de manejo de eventos, manejo de estado y sincronización con un backend en la nube (Firebase).

Recuerda la filosofía del curso: puedes apoyarte en herramientas de IA para escribir código, pero tú eres responsable de cada línea que entregas. En la defensa se te preguntará por qué tu código está escrito como está. Si no puedes explicarlo, no es tuyo.

2. Objetivos de aprendizaje

  • Construir una SPA en React usando componentes funcionales y hooks (useState, useEffect, y opcionalmente useContext / useReducer).
  • Aplicar buenas prácticas de manejo de eventos: handlers nombrados, prevención de comportamiento por defecto, validación antes de ejecutar acciones.
  • Aplicar el paradigma reactivo: la UI se deriva del estado, y el estado se sincroniza con Firestore mediante suscripciones en tiempo real (onSnapshot).
  • Integrar Firebase Authentication y Cloud Firestore en una aplicación real.
  • Manejar estados de carga, error y datos vacíos de forma explícita en la interfaz.

3. Descripción del proyecto

Debes construir XBank (o el nombre que quieras): una banca digital básica donde un usuario puede iniciar sesión, ver su saldo, transferir dinero a otros usuarios y revisar su historial de movimientos.

3.1 Requisitos funcionales (obligatorios)

RF1 · Autenticación

Registro e inicio de sesión con Firebase Authentication (email/contraseña). Un usuario no autenticado no puede ver nada más que el login/registro. Al registrarse, se crea su cuenta en Firestore con un saldo inicial de $100.000.

RF2 · Dashboard con saldo en tiempo real

Al entrar, el usuario ve su nombre y su saldo actual. El saldo se lee con onSnapshot: si cambia en Firestore (por ejemplo, porque otro usuario le transfirió), la UI se actualiza sola, sin refrescar.

RF3 · Transferencias

Formulario para transferir a otro usuario (por email o identificador). Validaciones mínimas: monto mayor a 0, saldo suficiente, no transferirse a sí mismo, destinatario existente. La transferencia descuenta al emisor, abona al receptor y registra el movimiento.

RF4 · Historial de movimientos

Lista de movimientos del usuario (enviados y recibidos), ordenada del más reciente al más antiguo, también en tiempo real. Cada movimiento muestra fecha, contraparte, monto y tipo (envío/recepción).

RF5 · Cerrar sesión

Botón de logout que limpia el estado y devuelve al login. Ojo: al cerrar sesión deben cancelarse las suscripciones activas.

3.2 Opcionales (bonificación, máx. +0,5 en la nota)

Bonus

  • Depósito/retiro simulado (botones que modifican el saldo con validación).
  • Filtro o búsqueda en el historial (por tipo, fecha o contraparte).
  • Modo oscuro persistente o diseño destacable (que se note intención, no solo Bootstrap por defecto).
  • Uso de useReducer + useContext para el estado global de la sesión.

4. Requisitos técnicos: aquí se juega la nota

4.1 Manejo de eventos

  • Handlers con nombre y responsabilidad clara (handleTransferSubmit, handleAmountChange), no funciones anónimas gigantes dentro del JSX.
  • Formularios controlados: el valor de cada input vive en el estado, no en el DOM.
  • event.preventDefault() en el submit de formularios. Si tu app recarga la página al transferir, algo hiciste mal.
  • Deshabilitar el botón de enviar mientras la operación está en curso (evitar doble submit = doble transferencia).
  • Validar antes de tocar Firestore. El usuario debe recibir feedback claro de qué está mal (mensaje visible, no solo console.log).

4.2 Programación reactiva

  • La UI es una función del estado: nada de manipular el DOM a mano (document.getElementById está prohibido dentro de componentes).
  • Saldo e historial se suscriben con onSnapshot, no con getDoc en un intervalo ni con botones de "actualizar".
  • Toda suscripción creada en un useEffect debe retornar su función de limpieza (unsubscribe). Fugas de memoria = descuento.
  • Arreglos de dependencias correctos en useEffect. Ni [] mágicos que esconden bugs, ni dependencias que generan loops infinitos.
  • Estados de carga y error explícitos: mientras llegan los datos se muestra un indicador, y si algo falla el usuario lo sabe.

4.3 Buenas prácticas generales

  • Componentes pequeños y con una responsabilidad. Un App.jsx de 400 líneas no es un componente, es un grito de auxilio.
  • Separar la lógica de Firebase en un módulo propio (ej. src/services/ o src/firebase/): los componentes no deberían importar Firestore directo por todos lados.
  • Nombres descriptivos en español o inglés, pero consistentes. Nada de data2, aux, cosa.
  • Las credenciales de Firebase van en variables de entorno (.env), y el .env va en el .gitignore. Incluye un .env.example con las claves vacías para que el profesor sepa qué configurar.
  • README con instrucciones para correr el proyecto y las credenciales de 2 usuarios de prueba.

4.4 Stack obligatorio

React 18+ (con Vite recomendado) · Firebase Authentication · Cloud Firestore. El estilo es libre (CSS puro, Tailwind, lo que quieras). No se permite usar librerías que resuelvan el problema por ti (ej. react-firebase-hooks): las suscripciones las escribes tú. No es necesario desplegar la app: se evalúa corriendo el proyecto en local con npm run dev.

4.5 Modelo de datos sugerido

users/{uid}        → { nombre, email, saldo }
movimientos/{id}   → { emisorUid, receptorUid, monto, fecha, descripcion }

Puedes proponer otro modelo si lo justificas en el README. Lo importante es que las consultas del historial y del saldo sean eficientes y en tiempo real.

5. Rúbrica de evaluación

Descuentos: la app no compila o no corre con las instrucciones del README (−2,0 puntos en la nota final) · credenciales de Firebase expuestas en el repositorio (−1,0) · manipulación directa del DOM dentro de componentes (−0,5) · suscripciones sin limpieza (−0,5).

6. Uso de IA

Puedes usar Claude, ChatGPT, Copilot o la herramienta que quieras. Es más: te recomiendo usarlas bien. Las reglas son las mismas de siempre:

  • Todo código que entregas lo debes poder explicar línea por línea en la defensa.
  • Si la IA te genera algo que no entiendes, tu trabajo es entenderlo antes de integrarlo, no pegarlo y rezar.
  • En el README incluye una sección breve (3–5 líneas) contando cómo usaste IA: qué le pediste, qué te sirvió y qué tuviste que corregir.

7. Formato de entrega

  • Repositorio GitHub (público o con acceso al profesor) con historial de commits real. Un solo commit "final" subido la última noche cuenta una historia que no te conviene contar.
  • README.md con: instrucciones de instalación y ejecución en local, 2 usuarios de prueba (email y contraseña), modelo de datos, y la sección de uso de IA.
  • El proyecto debe correr con npm install + npm run dev siguiendo el README. No se exige deploy.
  • Entrega en la plataforma del curso: link al repositorio, antes de la fecha límite indicada en clases.

Consejo final: parte por la autenticación y el saldo en tiempo real (RF1 y RF2). Si el día 3 ya tienes un usuario logueado viendo su saldo actualizarse solo, el resto es cuesta abajo. Si el día 6 recién estás configurando Firebase, se te vino la noche. Planifica.
